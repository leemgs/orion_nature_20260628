\section*{Methods}
\label{sec:methods}

\subsection*{Experimental framework: \textsc{Orion}}

\textsc{Orion} is a lightweight experimental framework for coordinating
accelerator memory, host DRAM, and NVMe storage as a unified hierarchy during
AI inference.
It is used here as a \emph{measurement instrument} rather than a production
serving engine: its role is to enable controlled traversal of the regime space
by making the latency components $T_{\mathrm{comp}}$, $T_{\mathrm{mem}}$,
$T_{\mathrm{swap}}$, and $T_{\mathrm{sync}}$ individually observable at
runtime.

\textsc{Orion} comprises three instrumented subsystems.
(1)~\textbf{GPU process allocation}: worker processes are pre-allocated and
pinned to reduce context-initialisation overhead and avoid contention-induced
stalls.
(2)~\textbf{NUMA-aware memory management}: allocations are bound to NUMA
domains co-located with the target GPU to reduce fragmentation and improve
locality.
(3)~\textbf{GPU-tailored swapping}: selective layer offloading and partial
reloads are implemented with asynchronous DMA transfers aligned to coarse
tensor-granularity blocks, enabling overlap of computation and cross-tier data
movement.

\subsection*{Regime probing methodology}

To expose regime-dependent behavior empirically, we vary two system
parameters independently: (i)~the fast-memory capacity ratio
$R_C = C_{\mathrm{fast}}/W$, controlled by adjusting the fraction of model
weights pinned in GPU HBM versus spilled to host DRAM; and
(ii)~the transfer-pressure ratio $R_B = B_{\mathrm{slow}}/D$, controlled by
throttling host-to-GPU PCIe bandwidth via hardware performance counters and
software rate limiters.
All other variables---model architecture, tokenisation, batch size, and
sequence length---are held constant within each probing sweep.

Regime boundaries are identified as the values of $R_C$ and $R_B$ at which the
dominant latency term changes by more than one standard deviation across five
independent runs, consistent with the analytical thresholds
$\theta_C \approx 0.5$ and $\theta_B \approx 0.4$ derived from the transfer
model.

\subsection*{Baselines and workloads}

Orchestration baselines span four categories: CPU offloading
(FlexGen~\cite{flexgen2023}, DeepSpeed-Inference~\cite{aminabadi2022deepspeedinference}),
GPU--CPU swapping (SwapAdvisor~\cite{huang2020swapadvisor}),
KV-cache management (vLLM~\cite{kwon2023pagedattention}),
and a PyTorch default serving baseline.
Language workloads use Llama-3~8B, GPT-J~6B, Llama-4~17B, and
Mixtral~8$\times$7B; the vision workload uses ViT-H/14.

\subsection*{Hardware platforms}

Primary experiments use two NVIDIA A100 (80~GB HBM) GPUs, 512~GB DDR5 host
DRAM, and an NVMe Gen4 SSD (software stack: PyTorch 2.4, CUDA 12.2, vLLM 0.5.3).
Cross-platform validation covers: Google TPU v4 pods (JAX 0.4.26), AWS
Inferentia2 endpoints (NeuronSDK 2.18), AMD MI250 GPUs (ROCm 6.0), and Intel
Xeon + Optane-PMem servers.
On all platforms, regime boundaries were identified using the same probing
methodology; reported transition points vary by $\leq 0.05$ from GPU-based
values.

\subsection*{Measurement protocol}

All latency measurements are averaged over $n = 5$ independent runs;
95\%~confidence intervals are reported throughout.
$T_{\mathrm{comp}}$ is isolated via CUDA events; $T_{\mathrm{mem}}$ is derived
from hardware performance counters (L2 cache misses, HBM bandwidth utilisation);
$T_{\mathrm{swap}}$ is measured via DMA transfer logs; and $T_{\mathrm{sync}}$
is measured as the residual after subtracting the other three components from
wall-clock time.
Energy consumption is measured using NVML power sampling at 100~ms intervals
and integrated over each inference request.

\subsection*{Ablation methodology}

To confirm that observed regime-dependent behavior is not driven by any single
\textsc{Orion} subsystem, we perform a component ablation by disabling each
subsystem independently (process allocation, NUMA binding, GPU-tailored
swapping) and repeating the full regime probing sweep.
Regime boundaries shift by $\leq 0.03$ across ablation variants, confirming
that the phase-like transitions are intrinsic to the hierarchical memory system
rather than artefacts of \textsc{Orion}'s implementation.

\subsection*{Regime classifier}

A depth-3 CART decision tree is trained on runtime statistics
(swap-to-computation ratio, cache hit rate, DMA utilisation) collected from
2,400 probing measurements covering all three regimes and five hardware
platforms.
The classifier is evaluated via stratified five-fold cross-validation; mean
accuracy is 93.4\% (s.d.\ 1.2\%).
Inference cost is $<$0.1~ms per classification step, negligible relative to
request latency.
